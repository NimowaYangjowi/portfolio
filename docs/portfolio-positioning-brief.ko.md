# 포트폴리오 포지셔닝 브리프

## 목표

이 포트폴리오의 목표는 한지우를 **테크니컬한 지식, 고객성공 경험, 실제 프로덕트 빌딩이 모두 있는 사람**으로 보이게 만드는 것이다.

개발자 용어로는 customer-facing technical operator이자 product builder다. 쉽게 말하면, 고객이 겪는 복잡한 문제를 듣고, 그 문제를 API와 데이터 흐름으로 정리하고, 실제로 사람이 쓸 수 있는 도구나 제품으로 만드는 사람이라는 뜻이다.

## 한 줄 포지셔닝

> 고객의 복잡한 업무 문제를 기술 요구사항, 운영 정책, 실제 제품 시스템으로 바꾸는 builder.

한국어 화면에서는 조금 더 자연스럽게 아래처럼 쓸 수 있다.

> 고객의 업무 문제를 API, 데이터, 운영 정책, 제품 화면으로 번역하고 실제로 굴러가는 시스템까지 만드는 사람입니다.

## 세 가지 증거 축

| 축 | 보여줄 것 | 근거 |
|---|---|---|
| Technical Knowledge | API, SDK, 데이터 구조, 결제/정산, worker, 장애 대응 흐름을 이해하고 설계할 수 있다. | Redprint의 Stripe/ledger/worker/admin 구조, AppsFlyer S2S Event Guide, SDK/API workflow 경험 |
| Customer Success | 고객의 숨은 업무 흐름을 발견하고, 글로벌 이해관계자와 실행 조건을 맞춰본 경험이 있다. | 글로벌/국내 enterprise accounts, APAC/SEA account operations, adoption bottleneck 해결 경험 |
| Product Building | 요구사항을 문서에서 끝내지 않고 실제 제품, 내부 콘솔, 운영 가이드로 만든다. | Redprint, OneLink Management Console, S2S implementation guide |

쉽게 말하면, 이 세 축은 각각 “기술을 안다”, “고객을 안다”, “직접 만든다”를 담당한다. 세 가지가 같이 보여야 포트폴리오가 평면적인 개발 프로젝트 소개에서 벗어난다.

## 사이트 섹션 매핑

| 섹션 | 역할 | 핵심 메시지 |
|---|---|---|
| Hero | 첫인상 형성 | 고객 문제를 기술과 제품으로 바꾸는 사람 |
| Proof Pillars | 강점 압축 | Technical Knowledge, Customer Success, Product Building |
| Customer-facing Case Studies | 고객성공 경험 증명 | 고객 대응이 아니라 workflow discovery와 technical translation을 해왔다 |
| Product Build Evidence | 실제 구현 증명 | Redprint는 화면, 데이터, 결제, 정산, 운영, worker가 연결된 시스템이다 |
| Operations Layer | 운영 감각 증명 | 운영자가 검증하고 조치할 수 있는 admin/worker 구조를 설계했다 |
| Source Trace | 신뢰 확보 | 주장마다 코드 또는 실제 사례 근거를 붙인다 |

## 핵심 사례별 사용법

### Redprint

- 역할: 실제 product building 증거.
- 강조점: marketplace domain architecture, media pipeline, payment/ledger, admin operations, worker recovery.
- 쉬운 설명: 매장 화면만 만든 것이 아니라 상품 등록, 결제, 정산, 고객지원, 운영실, 뒤에서 일하는 작업반까지 만든 사례다.

### OneLink Management Console

- 역할: 고객 업무를 운영 도구로 바꾼 증거.
- 강조점: API-heavy workflow, non-developer operator, create/update/get/delete 작업, 내부 콘솔.
- 쉬운 설명: 개발자가 아니어도 복잡한 링크 작업을 안전하게 처리할 수 있게 만든 도구 사례다.

### AppsFlyer S2S Event Guide

- 역할: technical translation 증거.
- 강조점: AFID-CUID mapping, S2S transmission condition, failure handling, security, monitoring.
- 쉬운 설명: 공식 문서만 보고는 고객이 구현하기 어려운 빈칸을 데이터 매핑과 실패 처리 기준으로 메운 사례다.

### 글로벌 / 국내 / SEA Enterprise

- 역할: customer success와 글로벌 실행 경험 증거.
- 강조점: multi-region stakeholder alignment, product adoption, usage bottleneck, enterprise account operations.
- 쉬운 설명: 큰 고객과 여러 지역이 얽힌 상황에서도 문제를 정리하고, 제품팀과 고객이 같이 움직일 수 있게 만든 경험이다.

## 문장 톤

피해야 할 톤:

- `고객사를 관리했습니다.`
- `API 문서를 작성했습니다.`
- `관리자 페이지를 구현했습니다.`
- `프로젝트를 만들었습니다.`

권장 톤:

- `고객의 업무 흐름을 API와 데이터 요구사항으로 번역했습니다.`
- `비개발자 운영자가 복잡한 API 작업을 안전하게 실행할 수 있는 콘솔로 바꿨습니다.`
- `정산, 원장, reconciliation을 연결해 운영자가 숫자의 원인을 추적할 수 있게 만들었습니다.`
- `사용자 화면 뒤에서 오래 걸리는 작업을 worker가 이어받고 복구할 수 있게 설계했습니다.`

쉽게 말하면, 이력서처럼 “무엇을 했다”만 쓰지 말고 “누구의 어떤 어려움이 줄었는지”까지 함께 말한다.

## 차별점

이 포트폴리오의 차별점은 “개발자처럼 만들 수 있다”와 “CS처럼 고객을 안다”를 따로 보여주는 것이 아니라, 두 가지가 만나는 지점을 보여주는 데 있다.

방문자가 가져가야 하는 인상은 아래와 같다.

1. 기술 구조를 이해한다.
2. 고객과 운영자의 실제 업무 언어를 안다.
3. 요구사항을 실행 가능한 제품 흐름으로 바꾼다.
4. 만든 뒤 운영, 복구, 검증까지 생각한다.

쉽게 말하면, 기획서만 쓰는 사람도 아니고, 화면만 만드는 사람도 아니고, 고객 문제를 끝까지 굴러가는 시스템으로 만드는 사람처럼 보여야 한다.
